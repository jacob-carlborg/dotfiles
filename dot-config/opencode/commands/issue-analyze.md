Analyze a GitHub issue against this codebase to identify questions that need answering before implementation can begin. Then post those questions as a comment on the issue.

The argument is: $ARGUMENTS

## Step 1: Fetch the issue

Follow these rules to determine which issue to fetch:

1. **If an argument is provided**, parse it as follows:
   - If it matches the pattern `owner/repo#NUMBER` (e.g. `rails/rails#12345`), fetch issue NUMBER from that specific repository.
   - If it is a plain number (e.g. `42`), fetch that issue number from the default repository (see rule 2).
   - If $ARGUMENTS is empty, present a list of issues to choose from (see rule 3).

2. **Determine the default repository** by running these git commands:
   - First try: !`git remote get-url upstream 2>/dev/null`
   - Fall back to: !`git remote get-url origin 2>/dev/null`
   - Extract the `owner/repo` from the remote URL (strip any `https://github.com/` prefix or `git@github.com:` prefix and `.git` suffix).

3. **If no issue number is given** ($ARGUMENTS is empty), list open issues from the default repository for the user to choose from:
   - Run `gh issue list --repo OWNER/REPO --state open --limit 20 --sort created --json number,title,createdAt --order desc` to get the 20 most recent open issues.
   - Present the list to the user using the interactive question tool (mcp_question) with each issue as an option, formatted as `#NUMBER: TITLE`.

4. **Fetch the selected issue** by running:
   ```
   gh issue view NUMBER --repo OWNER/REPO --json title,body,number,state,labels,assignees,milestone,url
   ```

5. Also fetch existing comments to avoid asking questions that have already been answered:
   ```
   gh issue view NUMBER --repo OWNER/REPO --json comments
   ```

## Step 2: Analyze the codebase

Thoroughly explore the codebase to understand how the requested change from the issue would fit in. Consider:

- **Architecture**: Which parts of the codebase are relevant? What patterns and conventions are used?
- **Scope**: What files, modules, or components would need to change?
- **Dependencies**: Are there dependencies or constraints that affect the approach?
- **Edge cases**: What edge cases or failure modes should be considered?
- **Testing**: What testing strategy would be appropriate?
- **Ambiguity**: What is unclear, underspecified, or could be interpreted multiple ways in the issue description?

## Step 3: Formulate questions

Based on the analysis, compile a list of specific, actionable questions that need to be answered before implementation can start. Focus on:

- Ambiguities in the requirements
- Design decisions that aren't specified
- Behavioral edge cases that aren't addressed
- Acceptance criteria that are missing or unclear
- Anything where the codebase suggests multiple valid approaches

Do NOT include questions that can be answered by reading the codebase. Only ask about things that require human input or product decisions.

If you have no questions, say so and explain why the issue is clear enough to implement directly.

## Step 4: Present findings to the user

Present to the user:
- A summary of the issue
- Your understanding of which parts of the codebase are affected
- The list of questions
- A draft of the GitHub comment you intend to post

Then ask the user for confirmation before posting using the interactive question tool (mcp_question). Offer "Post comment" and "Cancel" as options.

## Step 5: Post the comment

Once the user confirms, post the questions as a comment on the GitHub issue:

```
gh issue comment NUMBER --repo OWNER/REPO --body "COMMENT_BODY"
```

Format the comment clearly with a heading and numbered questions. Start the comment with a note that the questions were generated by an AI analysis of the codebase.

Do NOT make any file changes. This command only reads the codebase and posts a comment.
